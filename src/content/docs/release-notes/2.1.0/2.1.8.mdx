**Release Date:** 2026-06-25

**Release Type:** Minor

**Objective:** EKS upgrade from version 1.33 to 1.34, including control plane, addons, and node group.

**Focus:** EKS cluster version bump with updated addon versions and permanent node group version tracking via terraform.

**Expected downtime:** ~30 minutes (control plane + addons + rolling node replacement).

## What's New

- **EKS 1.34** — Control plane, addons, and node group upgraded from 1.33 to 1.34.
- **Updated addons:**

| Addon | 1.33 version | 1.34 version |
|---|---|---|
| kube-proxy | v1.33.0-eksbuild.2 | v1.34.6-eksbuild.13 |
| vpc-cni | v1.20.0-eksbuild.1 | v1.22.2-eksbuild.1 |
| coredns | v1.12.2-eksbuild.4 | v1.13.2-eksbuild.11 |

## Migration Steps

### 1. Plan and apply

This step upgrades the control plane first, then addons, then triggers rolling node replacement — terraform handles the order automatically.

```bash
cd obsrv-automation/terraform/aws

terraform plan  -var-file=vars/cluster_overrides.tfvars -target=module.eks
terraform apply -var-file=vars/cluster_overrides.tfvars -target=module.eks
```

Apply order (terraform handles this automatically):
1. EKS control plane → `~5 min`
2. Addons → `~2 min`
3. Node group rolling replacement → `~8-10 min per node`

### 2. Monitor node rollout

Each node is drained and replaced one at a time. Watch node status until all show the new version before proceeding to verify.

```bash
watch -n 10 'kubectl get nodes -o wide'
```

### 3. Verify

```bash
# Nodes — all must show v1.34.x
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.status.nodeInfo.kubeletVersion}{"\n"}{end}'

# No broken pods
kubectl get pods -A | grep -Ev "Running|Completed"
```

:::note
- For upgrades from **v2.1.x to v2.1.8**, upgrade only the **EKS cluster** (terraform apply targeting `module.eks`). No helm bundle changes required.
- When upgrading to release 2.1.8, use version 2.0.3 of enterprise-automation/obsrv-scripts-infy.
:::
